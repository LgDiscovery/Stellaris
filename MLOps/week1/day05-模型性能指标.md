# 一、核心模型性能指标（分类 / 回归两大任务核心指标）

模型性能指标是**量化模型预测效果**的工程化标准，不同任务（分类 / 回归）适配不同指标，是 MLOps 工程师判断模型是否可用、版本迭代的核心依据，核心指标分为两类：

### （一）分类任务指标（适用于预测 “类别” 的模型，如风控模型（是否风险用户）、推荐模型（是否点击）、图像分类（猫 / 狗））

1. **准确率（Accuracy）**：模型预测正确的样本数 / 总样本数，反映模型整体预测正确率；
2. **精确率（Precision，精准率）**：预测为正的样本中，实际为正的样本数 / 所有预测为正的样本数，反映 “模型预测的正样本里，真的对了多少”；
3. **召回率（Recall，查全率）**：实际为正的样本中，模型预测为正的样本数 / 所有实际为正的样本数，反映 “模型把真实的正样本，找出来了多少”；
4. **F1 值（F1-Score）**：精确率和召回率的**调和平均数**，公式为 F1=2×Precision+RecallPrecision×Recall，综合衡量精确率和召回率，避免单一指标的片面性；
5. **AUC（Area Under Curve）**：ROC 曲线下的面积，取值 0-1，反映模型对 “正 / 负样本” 的区分能力，AUC 越接近 1，区分能力越强；

### （二）回归任务指标（适用于预测 “连续数值” 的模型，如订单量预测、房价预测、销售额预测）

1. **MAE（平均绝对误差，Mean Absolute Error）**：所有样本预测值与真实值的绝对误差的平均值，公式为 MAE=n1∑i=1n∣yi−yi^∣，反映预测值与真实值的平均偏差，单位与预测目标一致，易理解；
2. **MSE（均方误差，Mean Squared Error）**：所有样本预测值与真实值误差的平方的平均值，公式为 MSE=n1∑i=1n(yi−yi^)2，对极端误差（异常值）更敏感，会放大偏差；
3. **RMSE（均方根误差，Root Mean Squared Error）**：MSE 的平方根，公式为 RMSE=MSE，还原误差的量纲，与 MAE 一样可直接反映偏差大小，是工程中最常用的回归指标；
4. **R²（决定系数）**：取值 0-1，反映模型对数据规律的拟合程度，R² 越接近 1，模型能解释的样本变化越多，拟合效果越好。

# 二、指标的工程意义：如何判断模型是否可用

工程视角下，指标并非 “越高越好”，而是**贴合业务需求、达到预设阈值、线上线下一致**，判断模型可用的核心原则为 3 点：

1. **指标达标业务阈值**：根据业务场景制定可落地的指标基准（如风控模型 AUC≥0.85、订单预测模型 RMSE≤50 单），指标达到 / 超过基准，才具备落地基础；
2. **指标无明显偏科**：单一指标无法反映模型全貌，如分类模型准确率 99%，但召回率 0%，实际无业务价值，需综合看多指标（如准确率 + 召回率 + F1）；
3. **线上线下指标一致**：训练 / 测试集的指标（线下）与线上真实业务数据的指标（线上）偏差≤预设范围（如 ±5%），避免模型 “线下效果好，线上失效”（如数据漂移导致的指标偏差）。

**工程落地小原则**：分类任务优先看**业务适配的精确率 / 召回率 + F1/AUC**，回归任务优先看**RMSE/MAE+R²**，同时结合业务实际损失判断（如指标达标但业务侧损失过高，模型仍不可用）。

# 三、准确率、召回率的适用场景区分

准确率是**整体正确率**，但在**样本不平衡场景**（如风控模型中，风险用户仅占 1%，正常用户 99%）会完全失效（模型全预测为正常用户，准确率 99%，但毫无业务价值），因此工程中极少单独用准确率判断模型，需结合**精确率 + 召回率**按业务场景选择核心优化目标，二者的适用场景严格区分：

### （一）优先优化**召回率**的场景：**漏判损失 > 误判损失**，要求 “不放过任何一个真实正样本”

核心诉求：把所有真实的正样本找出来，哪怕多预测一些假正样本，后续可通过人工 / 其他流程过滤，漏判会带来严重的业务损失。

**典型场景**：

* 金融风控：找风险用户，漏判一个高风险用户会导致平台资金损失，误判正常用户可通过人工复核排除；
* 医疗诊断：检测疾病，漏判患者会延误治疗，误判健康人群可通过二次检查确认；
* 反欺诈：识别欺诈订单，漏判欺诈订单会造成直接经济损失，误判正常订单可人工审核。

### （二）优先优化**精确率**的场景：**误判损失 > 漏判损失**，要求 “预测的正样本必须尽可能准”

核心诉求：模型预测的正样本要保证真实性，哪怕漏掉一些真实正样本，误判会带来用户体验 / 品牌损失。

**典型场景**：

* 商品推荐：推荐用户感兴趣的商品，精确率低会推荐无关商品，导致用户反感、卸载 APP，漏掉部分感兴趣商品对用户体验影响较小；
* 广告投放：精准触达潜在客户，精确率低会浪费广告预算，漏掉部分客户不会造成直接损失；
* 优质内容筛选：平台推荐精品内容，精确率低会推低质内容，影响平台口碑，漏掉部分精品内容影响有限。

### （三）准确率的适用场景：仅适用于**样本均衡**（正 / 负样本占比接近）、**漏判 / 误判损失相近**的简单场景

**典型场景**：图像分类（猫 / 狗样本各 50%）、手写数字识别（0-9 样本分布均匀），无明显的漏判 / 误判优先级，只需关注模型整体预测正确率。

# 四、MLOps 工程师关注这些指标的核心目的

MLOps 工程师的核心职责是**让模型稳定、高效、可持续地落地业务**，关注准确率、召回率、F1、MAE 等指标，并非单纯追求指标数值，而是为了实现**工程化的模型管理**，核心目的有 4 点：

1. **模型选型与版本迭代**：同一业务的不同模型版本（如 V1/V2/V3），通过指标量化效果，选择最优版本落地；后续模型迭代时，以指标为依据判断迭代是否有效（如迭代后 F1 值提升 5%，说明迭代成功）；
2. **把控模型落地的业务风险**：通过指标匹配业务需求，避免 “指标好看但业务无用” 的模型上线（如风控模型只看准确率，忽略召回率，导致大量风险用户漏判，引发业务损失）；
3. **监控模型线上退化**：将线下指标作为基准，实时监控线上模型的指标变化，若指标出现无理由的持续下跌（如召回率从 90% 跌至 60%），及时触发告警，排查是否存在数据漂移、概念漂移等问题；
4. **标准化模型交付与协作**：指标是量化的 “模型效果语言”，让算法工程师、MLOps 工程师、业务方达成共识，避免主观判断（如 “模型效果好” 无法落地，而 “模型 AUC≥0.88、召回率≥90%” 是可落地的交付标准）。

简单来说，MLOps 工程师把这些指标作为**模型全生命周期的 “核心监控标尺”**，从模型研发、落地到线上运维，通过指标实现模型效果的量化、标准化、可监控。
